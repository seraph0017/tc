## 环节4.1 讲授A：Skill系统设计（0:15 - 0:45）

### Part 1：无约束代码生成的四大问题（0:15 - 0:25）

[切换幻灯片到4-5]

[语气：讲故事] 回顾一下最初的代码生成方式。你说"帮我写一个Python函数，计算斐波那契数列"，AI"唰"一下就给你了。对于这种简单任务，直接生成完全没问题。

但当需求复杂度上升——比如"帮我写一个用户管理系统"——问题就来了。

[切换幻灯片到4-6，展示四大问题]

**问题一：缺乏结构。** AI可能把所有代码写在一个文件，也可能拆成10个文件。不同次生成的结果结构完全不同。

**问题二：质量不稳定。** 今天生成的代码有异常处理和类型注解，明天生成的就没有。没有标准保证最低质量线。

**问题三：难以维护。** 没有统一的命名规范、注释规范、测试覆盖。三个月后回来看，比看别人的代码还费劲。

**问题四：无法复现。** "上次那段代码效果很好，再来一次"——几乎不可能得到相同质量的输出，因为没记录"上次怎么做的"。

[互动] 大家有没有过这种经历：让AI写了一段代码，当时觉得挺好，过了一个月再看——"这啥？？"

[停顿3秒，等待反应]

[语气：笑] 我猜很多人都有。问题不在AI不行，而在于我们没给它画好框架。

[切换幻灯片到4-7]

OpenClaw团队从中得到了一个核心洞察——**"约束即自由"**。给AI施加明确的结构化约束，反而能释放它的生产力。这就像软件工程中的设计模式——表面上是限制，实际上是最佳实践的沉淀。

[停顿1秒] 那怎么实现这个"约束"？答案就是Skill系统。

### Part 2：Skill系统架构（0:25 - 0:37）

[切换幻灯片到4-8]

#### 目录结构

Skill系统的核心设计理念是：**把代码生成的最佳实践封装成可复用的"技能包"**。每个Skill是一个独立的目录。

[切换幻灯片到4-9，展示目录结构]

大家看这个结构：

```
skills/
├── coding-agent/              # 通用编码技能
│   ├── SKILL.md               # 技能定义（核心文件）
│   ├── templates/             # 代码模板
│   └── scripts/               # 辅助脚本
├── test-driven-development/   # 测试驱动开发技能
└── systematic-debugging/      # 系统化调试技能
```

每个Skill就像一本"操作手册"——告诉AI面对某类任务时，该按什么步骤、什么标准来执行。

而这本操作手册的灵魂，就是SKILL.md。

#### SKILL.md的六要素

[切换幻灯片到4-10，展示SKILL.md六要素]

SKILL.md必须包含六个要素。我逐个讲，每个都很重要。

[语气：逐条推进，像讲合同条款但更生动]

**要素一：适用场景** ——"什么时候用"。比如coding-agent适用于构建新功能、审查PR、重构代码库。

**要素二：输入要求** ——"需要提供什么"。功能需求是必须的，技术栈和性能要求是可选的。

**要素三：输出格式** ——"产出长什么样"。比如每个文件必须包含文件头注释、类型注解、文档字符串。

[语气：加重] 这个要素对质量一致性贡献最大。它直接定义了"什么是合格的输出"，相当于设定了质量基线。

**要素四：工作流程** ——"按什么步骤做"。分析需求、设计架构、编写测试、实现代码、验证运行、代码审查——六步，顺序不能乱。

**要素五：示例** ——"做好是什么样"。给AI看一个完整案例，它就知道方向。

**要素六：常见陷阱** ——"容易错在哪"。比如"不要跳过测试""不要把所有逻辑放一个文件"。

[切换幻灯片到4-11，展示六要素设计逻辑表格]

这六个要素不是随便排的。它们构成了一个完整的认知闭环——从"什么时候用"到"容易错在哪"，覆盖了执行一个任务的全部决策点。

[互动] 我来考大家：如果你要为"数据库迁移"这个任务设计Skill，"常见陷阱"部分应该写什么？给你们10秒钟想两个。

[停顿10秒]

[语气：引导] 忘记备份数据就执行迁移？没有编写回滚脚本？在生产环境直接跑未经测试的迁移？大表加列没考虑锁表？——这些全是血泪教训。写进SKILL.md，AI就不会犯同样的错。

### Part 3：模板机制（0:37 - 0:45）

[切换幻灯片到4-12]

除了SKILL.md，Skill目录里还有`templates/`文件夹。模板是什么？就是预定义的代码骨架。

[语气：打比方] 你去买房，开发商不会让你从地基开始自己盖。他给你一个户型——客厅在哪、卧室在哪、厨房在哪——已经规划好了，你在这个框架里装修。模板就是代码的"户型图"。

比如FastAPI项目的模板会预定义：项目目录结构、每个文件的骨架代码、配置文件的格式。AI在生成代码时，不是从零开始，而是"填充"这些模板。这就保证了每次生成的项目结构一致。

[停顿1秒]

好，Skill系统的设计讲完了。接下来的问题是：有了Skill，代码是怎么一步步生成出来的？

## 环节4.2 讲授B：代码生成六步工作流（0:45 - 1:15）

[切换幻灯片到4-13，展示六步工作流全景图]

### 六步工作流总览

[语气：清晰、有节奏] 从"用户请求"到"可运行代码"，一共六步：**意图分析、技能选择、架构设计、代码生成、自动验证、迭代优化。**

我们逐步来看。

### Step 1：意图分析（0:47 - 0:52）

[切换幻灯片到4-14]

这一步的核心任务是：**把模糊的自然语言请求，转化为结构化的任务描述。**

[切换幻灯片到4-15，展示意图分析伪代码]

比如用户说："帮我创建一个用户管理的REST API，使用FastAPI，包含JWT认证。" 意图分析会输出一个结构化描述：任务类型是"新建应用"，复杂度"中等"，技术栈是FastAPI+SQLAlchemy+JWT，功能需求包括用户CRUD、JWT认证、数据持久化。如果有模糊点，系统会先追问再继续。

[语气：强调] 这一步做不好，后面全白费。就像医生问诊——没搞清楚哪里不舒服，开再好的药也没用。

### Step 2：技能选择（0:52 - 0:55）

[切换幻灯片到4-16]

根据意图分析的结果，选择最匹配的Skill。在这个例子中，"新建应用"任务匹配的是`coding-agent`技能。如果没有完美匹配的Skill呢？用通用的coding-agent作为兜底。

### Step 3：架构设计（0:55 - 1:02）

[切换幻灯片到4-17]

[语气：加重] 这一步是整个流程的**质量分水岭**。好的架构设计意味着模块职责清晰、接口定义明确。

[切换幻灯片到4-18，展示项目结构]

对于我们的FastAPI案例，架构设计的输出是这样一个项目结构：

```
├── main.py          # 应用入口
├── models.py        # 数据模型（SQLAlchemy ORM）
├── schemas.py       # 接口模型（Pydantic请求/响应）
├── auth.py          # 认证逻辑（JWT生成/验证）
├── crud.py          # 数据操作（数据库读写封装）
├── routers/
│   └── users.py     # 用户路由（REST API端点）
└── tests/
    └── test_users.py # 测试文件
```

7个文件，每个文件只做一件事。models.py只管数据库表结构，schemas.py只管API数据格式，auth.py只管认证。修改数据库结构不用碰认证逻辑，修改API格式不用碰数据库模型。

[语气：点题] 这就是"单一职责原则"在实际项目中的体现。

### Step 4：代码生成（1:02 - 1:05）

[切换幻灯片到4-19]

按照Skill规范和架构设计，逐文件生成代码。具体代码我们下半场的案例分析环节详细走读。

### Step 5：自动验证（1:05 - 1:10）

[切换幻灯片到4-20]

[语气：严肃] 这是最容易被跳过的一步，也是最不应该被跳过的一步。

自动验证分三个层次：第一层，语法检查——用`python -m py_compile`确保代码没有语法错误。第二层，类型检查——用`mypy`验证类型注解是否正确。第三层，测试运行——用`pytest`验证功能是否符合预期。

[互动] 来，我出个思考题：如果跳过Step 5，最可能的后果是什么？

[停顿5秒]

[语气：揭示] 后果是——你拿到的代码看起来"完整"，但运行时才爆雷。自动验证是代码质量的最后一道防线。没有这道防线，前面所有步骤的努力都可能打水漂。

### Step 6：迭代优化（1:10 - 1:12）

[切换幻灯片到4-21]

如果Step 5发现问题，回到Step 4修复代码，然后再次验证。这个循环通常1到3轮就能收敛。如果3轮还没收敛——标记问题，请求人工介入。不要让AI无限循环。

### 异常处理总览（1:12 - 1:15）

[切换幻灯片到4-22，展示异常处理表格]

每一步都可能出异常。我快速过一下：意图分析阶段需求模糊就追问；架构设计阶段发现需求超出单服务范围就建议拆分；代码生成阶段上下文窗口不足就分文件逐个生成；自动验证测试失败就分析原因回去修复。

[语气：总结性] 整个六步工作流，本质上就是一个"规划—执行—验证—修正"的闭环。跟我们人类写代码的过程是一样的，只不过每一步都被系统化了。

## 环节4.3 案例分析：FastAPI用户管理API（1:25 - 1:55）

[切换幻灯片到4-23]

### 案例背景

[语气：切换到代码讲解模式] 好，回来。我们现在用一个真实案例把六步工作流从头走到尾。

用户请求是："帮我创建一个用户管理的REST API，使用FastAPI，包含JWT认证。"

意图分析和技能选择刚才已经讲过了，架构设计也看过了。现在我们直接进入核心——逐文件看生成的代码。

### 文件1-2：models.py 和 schemas.py（1:27 - 1:35）

[切换幻灯片到4-24，展示models.py代码]

[语气：像做代码review]

先看models.py——只做一件事：定义数据库表结构。

重点看第9-10行。email字段加了`unique`和`index`两个约束。为什么？`unique`保证不会有重复注册，`index`加速按邮箱查询——登录时需要。每个约束都有业务含义，不是随便加的。

再看第15行——字段名是`hashed_password`，不是`password`。**永远不存明文密码。** 即使数据库泄露，密码也不会直接暴露。

[切换幻灯片到4-25，展示schemas.py代码]

[互动] 为什么models.py和schemas.py要分两个文件？它们不都是在定义"用户"吗？

[停顿5秒]

[语气：揭示] 关键区别：models.py面向**数据库**（内部），schemas.py面向**API**（外部）。密码就是最好的例子——`UserCreate`里是明文`password`用于接收输入，models里是`hashed_password`用于存储。而`UserResponse`**不包含**任何密码字段——只返回客户端需要的信息，不多不少。这叫"最小暴露原则"。

### 文件3：认证逻辑 auth.py（1:35 - 1:42）

[切换幻灯片到4-26，展示auth.py代码]

[语气：严肃] 这是安全核心。三个要点。

第一，第7行`SECRET_KEY = "your-secret-key"`——**生产环境必须从环境变量读取，绝不能硬编码。** 教学演示才这么写。

第二，为什么用bcrypt而不是SHA256做密码哈希？bcrypt专门为密码设计，内置"盐"防彩虹表攻击，还有"慢哈希"机制让暴力破解极其耗时。"慢"在这里是优点。

第三，JWT令牌设了过期时间——默认15分钟。一旦令牌泄露，危害被限制在时间窗口内。

### 文件4-6：crud.py、routers/users.py、main.py（1:42 - 1:52）

[切换幻灯片到4-27，展示crud.py代码]

crud.py封装所有数据库读写。为什么不直接在路由里操作数据库？三个原因：职责分离、逻辑复用、方便测试。

[切换幻灯片到4-28，展示routers/users.py代码]

路由文件的设计原则是"薄"——`create_new_user`只做三件事：检查邮箱是否已注册、调用CRUD层创建、返回响应。`Depends(get_db)`是FastAPI的依赖注入：自动创建数据库会话、注入参数、用完关闭。

[切换幻灯片到4-29，展示main.py和依赖关系图]

main.py只做"组装"——初始化应用、注册中间件、挂载路由。越简洁越好。

### 文件间依赖关系（1:52 - 1:55）

[语气：总结性] 最后看依赖关系全景：路由层依赖业务层（crud/auth），业务层依赖模型层（models/schemas）。**依赖方向单一**，反向依赖绝不允许。这是可维护性的根基。

[互动] 走读完7个文件了。回想一下——没有Skill系统的约束，AI大概率会把这些逻辑全揉进一个文件。那就是开头对比演示里左边的情况。

## 环节4.5 讲授C：测试生成与写作系统（2:25 - 2:45）

### Part 1：测试驱动生成（2:25 - 2:35）

[切换幻灯片到4-31]

[语气：承上启下] 好，练习做完了。现在我们来看Skill系统的两个扩展应用。第一个是测试驱动的代码生成。

在传统TDD中，测试是写给人看的需求规约。在AI代码生成中，测试还有一个更重要的角色——**自动验证生成结果的正确性**。

[语气：打比方] 没有测试的代码生成，就像没有质检的工厂——你只能在客户投诉时才知道产品有问题。

[切换幻灯片到4-32，展示测试代码关键片段]

快速看一下FastAPI案例的测试文件。我挑三个设计要点讲。

**第一，内存数据库。** 第9行——`SQLALCHEMY_DATABASE_URL = "sqlite:///:memory:"`。为什么用内存数据库？三个好处：速度快，不需要磁盘IO；隔离性好，每次测试都是全新的；无残留，测试结束自动清除。

**第二，依赖覆盖。** 第22行——`app.dependency_overrides[get_db] = override_get_db`。这行代码把生产数据库替换成了测试数据库。不修改任何业务代码，只替换依赖，就能切换到测试环境。这就是依赖注入的威力。

**第三，正常路径和异常路径都要测。** `test_create_user`测正常创建，`test_create_duplicate_user`测重复邮箱注册。好的测试不只是证明"能用"，还要证明"该报错的时候会报错"。

[互动] 来，思考一个问题：为什么测试数据库要用`StaticPool`连接池？如果不用会怎样？

[停顿5秒]

[语气：解释] SQLite内存数据库有个特性：每个连接看到的是独立的数据库。如果用默认连接池，不同请求可能获得不同连接，一个请求写入的数据另一个请求看不到。`StaticPool`确保所有请求共享同一个连接，从而共享同一个内存数据库。

### Part 2：写作Skill系统（2:35 - 2:45）

[切换幻灯片到4-33]

[语气：视野拓展] 第二个扩展应用更有意思——Skill系统不只能用于代码生成，还能用于**写作**。

[切换幻灯片到4-34，展示写作Skill的输入输出规范]

原文第7章展示了一个"技术博客写作"Skill。看它的输入输出规范：输入是topic（主题）、target_audience（目标读者）、word_count（目标字数）。输出的format是Markdown，sections是固定的7个部分：title、tldr、problem、solution、results、trade-offs、conclusion。

[语气：关键洞察] 注意——输出的sections是**固定的**。这看似限制了灵活性，但实际上保证了每篇文章都覆盖读者最关心的内容。如果让AI自由决定结构，它可能跳过"trade-offs"部分，让文章缺乏客观性；可能跳过"problem"部分，让读者不知道为什么要看这篇文章。

[切换幻灯片到4-35，展示五阶段写作流程]

写作Skill也有自己的工作流，五个阶段：需求分析、大纲生成、内容生成（逐章节）、代码验证（如果文中有代码片段）、格式优化。

[切换幻灯片到4-36，展示代码Skill vs 写作Skill对比表]

对比一下：输入不同、输出不同、验证方式不同。但共性是什么？**都需要明确的输入规范、固定的输出结构、可执行的工作流程。**

[语气：升华] 这告诉我们一个重要的道理：Skill系统的本质是**将隐性知识显性化**。资深开发者"知道"好代码该什么样——通过coding-agent Skill显性化；优秀作者"知道"好文章该什么结构——通过writing Skill显性化。显性化之后，AI就能按这些标准执行，输出质量从"看运气"变成"有保障"。

### 三句话总结

好，到了收尾时间。今天三个小时，我们学了什么？我用三句话总结：

[语气：缓慢、清晰，每句话之间停顿2秒]

**第一，无约束的AI代码生成有四大问题——结构不一致、质量不稳定、难以维护、无法复现。** 这些问题的根源不是AI能力不行，而是我们没给它明确的标准。

**第二，Skill系统通过SKILL.md的六要素标准化定义，加上六步代码生成工作流，把代码生成从"碰运气"变成了"有章法"。** 其中"输出格式"要素和"自动验证"步骤是质量保障的两个关键锚点。

**第三，Skill系统具有通用性——从代码到写作到数据分析，任何需要AI结构化输出的场景都适用。** 它的本质是将隐性知识显性化，将最佳实践沉淀为可复用的规范。

[停顿2秒]

一句话概括今天的核心哲学——

[切换幻灯片到4-38，大字展示"约束即自由"]

**约束即自由。** 给AI的约束越明确，它的输出越稳定、越可靠、越有价值。这不只适用于AI，也适用于我们自己的工作——有清晰的流程和标准，反而能让你把精力放在真正需要创造力的地方。
