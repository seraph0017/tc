### 三层配置优先级

```
优先级（低 → 高）：

+-------------------+
| 环境变量           |  ← 最高：紧急覆盖，不改文件不重启
| OPENCLAW_TRADING_  |

+-------------------+
        ↑ 覆盖
+-------------------+
| 配置文件 main.yml  |  ← 中间：针对特定部署
+-------------------+
        ↑ 覆盖
+-------------------+
| 代码中的默认值      |  ← 最低：保证系统能启动
+-------------------+
```

> 原则：越具体越优先

## [CODE] Slide 6-1e

### ConfigLoader -- 三层加载逻辑

```python
class ConfigLoader:
    DEFAULTS = {
        'environment': 'development',
        'memory': {'backend': 'file'},          # <-- 重点
    }

    def __init__(self, config_path: str = None):
        self.config = self.DEFAULTS.copy()       # 第1层

        if config_path and os.path.exists(config_path):
            with open(config_path, 'r') as f:
                file_config = yaml.safe_load(f)
            self._deep_merge(self.config, file_config)  # 第2层

        self._apply_env_overrides()              # 第3层 <-- 重点
```

## [DIAGRAM] Slide 6-1h

### 配置热加载 vs 重启生效

```
适合热加载（修改后立即生效）：
  报警阈值 / 日志级别 / 通知通道 / 模型选择
  --> 影响"下一次操作的行为"

必须重启才能生效：
  数据库连接参数 / 端口号 / Agent数量
  --> 影响"已经创建的资源"

热加载原理：
  ConfigWatcher 每30秒检查文件修改时间
  current_modified > last_modified --> reload_callback()
```

## [CODE] Slide 6-2b

### CronManager -- 任务注册与状态跟踪

```python
class CronManager:
    def add_task(self, name: str, schedule: str,
                 script: str, agent: str = "main"):
        task = {
            'id': self._generate_id(),
            'name': name,
            'schedule': schedule,           # <-- 重点
            'script': script,
            'agent': agent,
            'enabled': True,
            'last_run': None,               # 上次执行时间
            'last_status': None             # <-- 重点
        }
        self.tasks.append(task)
```

## [DIAGRAM] Slide 6-2d

### 任务设计三原则

```
原则1：错开执行时间
  22:00 量化复盘
  22:55 策略复盘          ← 间隔55分钟，避免资源争抢

原则2：监控高频，业务低频
  每小时 --> 系统健康检查（问题早发现）
  每天   --> 业务复盘（足够了）

原则3：关键任务有兜底
  "Cron失败监控" 专门监控其他任务是否正常
                   ← 监控的监控
```

## [QUESTION] Slide 6-3c

> 对应讲课稿 [互动]：为什么磁盘阈值（80%）低于内存阈值（90%）？

# 磁盘80% vs 内存90%

**为什么磁盘要更早报警？**

磁盘满 --> 数据库损坏、日志丢失，恢复慢

内存满 --> 重启服务即可释放

> 后果越严重 --> 报警越早

## [TABLE] Slide 6-3d

### 五项检查详解

| 检查项 | 超标后果 | 处理方式 |
|--------|---------|---------|
| 磁盘 80% | 日志无法写入，DB损坏 | 清理旧日志/扩容 |
| 内存 90% | 服务被OOM杀死 | 重启服务/排查泄漏 |
| API 5000ms | 超时失败，体验差 | 检查网络/切换备用API |
| 心跳 2x间隔 | Agent可能崩溃 | 检查日志并重启 |
| 连续失败 3次 | 定时任务停工 | 检查脚本/依赖 |

## [CODE] Slide 6-3e

### SystemMonitor -- 阈值配置与报警

```python
class SystemMonitor:
    def __init__(self):
        self.alert_channels = ['feishu']
        self.thresholds = {
            'disk_usage': 80,           # <-- 重点
            'memory_usage': 90,
            'failed_tasks': 3,
            'api_latency': 5000         # <-- 重点
        }

    def check_health(self) -> Dict:
        checks = {
            'disk': self._check_disk(),
            'memory': self._check_memory(),
            'gateway': self._check_gateway(),
        }
        alerts = [c for c in checks.values()
                  if not c['healthy']]    # <-- 重点
        if alerts:
            self._send_alert(alerts)
```

```

滚动更新：

                                    |

优势：零停机 / 问题即时发现 / 自动回滚
```
