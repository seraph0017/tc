## 页 15/30 [CONCEPT]

# 智能路由的工作方式

```
用户的请求进来
      ↓
  🔀 路由系统判断
      ↓
  ┌─────────────────────────────────┐
  │ "帮我debug代码"→ Claude(代码)  │
  │ "总结100页报告"→ Gemini(长文)  │
  │ "今天几号？"   → mini(快速)    │
  │ "设计产品方案" → Kimi(主力)    │
  └─────────────────────────────────┘
```

## 页 16/30 [CODE-BILINGUAL]

# 路由规则 = "分诊手册"

| 路由配置 | 白话翻译 |
|---------|---------|
| `pattern: "代码\|编程\|debug"` | 如果提到代码相关词 |
| `model: coding` | → 分给代码专家 |
| `pattern: "总结\|摘要"` | 如果提到总结相关词 |
| `model: long_context` | → 分给长文档专家 |
| `pattern: "快速\|简单"` | 如果是简单问题 |
| `model: fast` | → 分给快速模型 |
| `default: primary` | 都不匹配 → 用主力模型 |

# 不只看类型，还判断"难度"

```
分诊台护士不只看"什么科"
还要判断"严重程度"：

  ┌─────────────────────────┐
  │  复杂度评估              │
  │                           │
  │  问题长不长？             │
  │  有没有专业术语？         │
  │  需不需要深度思考？       │
  │           ↓               │
  │  简单(<0.3) → mini       │
  │  中等(0.3-0.8) → 普通   │
  │  复杂(>0.8) → 最强       │
  └─────────────────────────┘
```

> 同样是"代码问题"
> "Hello World" vs "重构万行系统" → 难度天差地别

## 页 22/30 [DIAGRAM]

# "选模型"三步法

```
第①步：定义任务类型
  简单对话 → 轻量模型
  内容生成 → 中等模型
  代码开发 → 代码专用
  长文档   → 大上下文
  复杂推理 → 最强模型

第②步：确定优先级
  速度优先(客服) → 用快速模型
  质量优先(报告) → 用最强模型
  成本优先(批量) → 用便宜模型

第③步：考虑预算
  < $50/月  → 全用轻量
  $50-500   → 智能路由混合
  > $500    → 更多场景用强力
```
